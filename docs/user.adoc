== Introduction

This guide explains how to use the JobRunr Control dashboard to schedule, configure, and monitor jobs.
The dashboard provides a user-friendly interface for managing background jobs without requiring direct code changes.

=== Accessing the Dashboard

The JobRunr Control dashboard is available at:

----
http://your-server:port/q/jobrunr-control
----

=== Security Setup

All JobRunr Control endpoints and the embedded JobRunr Pro Dashboard are protected by
authentication and role-based authorization using Quarkus OIDC (OpenID Connect).

==== Authentication Model

Quarkus is configured as a **Backend-for-Frontend (BFF)**. The browser never handles
tokens directly:

. The user navigates to `/q/jobrunr-control` — if not authenticated, Quarkus redirects to Keycloak.
. After login, Keycloak redirects back and Quarkus stores the token in an encrypted **session cookie**.
. All subsequent browser requests (Web UI, embedded JobRunr Pro Dashboard) carry this cookie — no bearer token is needed.
. External systems (CI/CD, automation) authenticate with a **bearer token** (OIDC Client Credentials) sent in the `Authorization` header.

==== Protected Endpoints

[cols="2,2,2"]
|===
|Endpoint |Authentication |Required Role(s)

|`/q/jobrunr-control`, `/q/jobrunr-control/*`
|OIDC session cookie
|Any authenticated user

|`/q/jobrunr`, `/q/jobrunr/*` (JobRunr Pro Dashboard)
|OIDC session cookie
|Any authenticated user

|`/q/jobrunr/api/*` — GET, OPTIONS, HEAD
|OIDC session cookie
|`viewer`, `configurator`, or `admin`

|`/q/jobrunr/api/*` — POST, PUT, DELETE, PATCH
|OIDC session cookie
|`admin` only

|`/q/jobrunr-control/api/*` — GET, OPTIONS, HEAD
|Bearer token
|`api-reader`, `api-executor`, or `admin`

|`/q/jobrunr-control/api/*` — POST, PUT, DELETE, PATCH
|Bearer token
|`api-executor` or `admin`
|===

[IMPORTANT]
====
**JobRunr Pro Dashboard Security**

JobRunr Pro's embedded dashboard mode does not support its own OpenID authentication
(see https://www.jobrunr.io/en/guides/authentication/openid-authentication/).
Protection is enforced entirely at the **Quarkus HTTP layer** via OIDC session cookies.

The extension automatically registers `JobRunrDashboardUserContextFilter`, which sets
the internal JobRunr user context after Quarkus has already authenticated the request.
This is required to prevent the dashboard from showing "You do not have access".
====

==== Deployment Modes

===== Production with Keycloak (Recommended)

Set up Keycloak with the `jobrunr` realm and configure `application.properties`:

[source,properties]
----
# Default OIDC tenant — Authorization Code Flow for the Web UI and JobRunr Dashboard
quarkus.oidc.auth-server-url=https://your-keycloak.com/realms/your-realm
quarkus.oidc.client-id=jobrunr-control
quarkus.oidc.credentials.secret=${OIDC_CLIENT_SECRET}
quarkus.oidc.application-type=web-app
# Must be a strong random value (32+ characters) — override via environment variable
quarkus.oidc.token-state-manager.encryption-secret=${SESSION_ENCRYPTION_SECRET}
# Keycloak stores realm roles at realm_access/roles in the access token
quarkus.oidc.roles.source=accesstoken
quarkus.oidc.roles.role-claim-path=realm_access/roles
quarkus.oidc.token.refresh-expired=true
quarkus.oidc.logout.path=/q/jobrunr-control/logout
quarkus.oidc.logout.post-logout-path=/q/jobrunr-control/scheduled

# Bearer tenant — validates bearer tokens for REST API service accounts only
# Named 'bearer', not 'api' — see note below
quarkus.oidc.bearer.tenant-paths=/q/jobrunr-control/api/*
quarkus.oidc.bearer.auth-server-url=https://your-keycloak.com/realms/your-realm
quarkus.oidc.bearer.client-id=jobrunr-control
quarkus.oidc.bearer.credentials.secret=${OIDC_CLIENT_SECRET}
quarkus.oidc.bearer.application-type=service
quarkus.oidc.bearer.roles.source=accesstoken
quarkus.oidc.bearer.roles.role-claim-path=realm_access/roles

# HTTP auth policies
quarkus.http.auth.permission.jobrunr-control.paths=/q/jobrunr-control,/q/jobrunr-control/*
quarkus.http.auth.permission.jobrunr-control.policy=authenticated
quarkus.http.auth.permission.jobrunr-dashboard.paths=/q/jobrunr,/q/jobrunr/*
quarkus.http.auth.permission.jobrunr-dashboard.policy=authenticated
quarkus.http.auth.permission.jobrunr-control-api-read.paths=/q/jobrunr-control/api/*
quarkus.http.auth.permission.jobrunr-control-api-read.policy=api-reader-policy
quarkus.http.auth.permission.jobrunr-control-api-read.methods=GET,OPTIONS,HEAD
quarkus.http.auth.permission.jobrunr-control-api-write.paths=/q/jobrunr-control/api/*
quarkus.http.auth.permission.jobrunr-control-api-write.policy=api-executor-policy
quarkus.http.auth.permission.jobrunr-control-api-write.methods=POST,PUT,DELETE,PATCH
quarkus.http.auth.permission.jobrunr-api-read.paths=/q/jobrunr/api/*
quarkus.http.auth.permission.jobrunr-api-read.policy=viewer-policy
quarkus.http.auth.permission.jobrunr-api-read.methods=GET,OPTIONS,HEAD
quarkus.http.auth.permission.jobrunr-api-write.paths=/q/jobrunr/api/*
quarkus.http.auth.permission.jobrunr-api-write.policy=admin-policy
quarkus.http.auth.permission.jobrunr-api-write.methods=POST,PUT,DELETE,PATCH

# Role policies
quarkus.http.auth.policy.admin-policy.roles-allowed=admin
quarkus.http.auth.policy.viewer-policy.roles-allowed=viewer,configurator,admin
quarkus.http.auth.policy.api-executor-policy.roles-allowed=api-executor,admin
quarkus.http.auth.policy.api-reader-policy.roles-allowed=api-reader,api-executor,admin
----

[IMPORTANT]
====
**Bearer tenant must be named `bearer`, not `api`.**

Quarkus OIDC has a convention-based fallback that matches tenant IDs against URL path
segments. A tenant named `api` would intercept all requests containing `/api/` in the
path — including the JobRunr Pro Dashboard's XHR calls to `/q/jobrunr/api/*` — and
reject them because no bearer token is present in the browser request.
====

===== Keycloak Realm Setup

Use the provided `keycloak-realm.json` export in the example project, or configure manually:

. Create a realm (e.g. `jobrunr`)
. Create a client `jobrunr-control` with:
** _Client authentication_: enabled
** _Valid redirect URIs_: `http://your-server:port/*`
** _Service accounts_: enabled (for API bearer token flow)
. Create the following **realm roles** (not client roles):
** `viewer`, `configurator`, `admin` (for Web UI users)
** `api-reader`, `api-executor` (for API service accounts)
. Assign roles to users and service accounts as appropriate.

[NOTE]
====
Roles must be **realm roles**, not client roles. Quarkus reads them from
`realm_access/roles` in the Keycloak access token. Client roles (under
`resource_access/<client>/roles`) are not read by the configured `role-claim-path`.
====

===== Production without OIDC (Closed Networks)

For internal deployments where no identity provider is available:

[source,properties]
----
quarkus.oidc.enabled=false
quarkus.http.auth.permission.jobrunr-control.paths=/q/jobrunr-control,/q/jobrunr-control/*
quarkus.http.auth.permission.jobrunr-control.policy=permit
quarkus.http.auth.permission.jobrunr-dashboard.paths=/q/jobrunr,/q/jobrunr/*
quarkus.http.auth.permission.jobrunr-dashboard.policy=permit
quarkus.http.auth.permission.jobrunr-control-api-read.paths=/q/jobrunr-control/api/*
quarkus.http.auth.permission.jobrunr-control-api-read.policy=permit
quarkus.http.auth.permission.jobrunr-control-api-write.paths=/q/jobrunr-control/api/*
quarkus.http.auth.permission.jobrunr-control-api-write.policy=permit
----

The built-in `JobRunrControlRoleAugmentor` automatically grants the `admin` role to all
requests when OIDC is disabled.

[WARNING]
====
When OIDC is disabled, the application performs **no authentication**. Protect access
using network-level controls (firewall, VPN, reverse proxy with IP restrictions).
====

==== Obtaining a Bearer Token for REST API Access

External systems authenticate using OIDC Client Credentials:

[source,bash]
----
TOKEN=$(curl -s -X POST \
  https://your-keycloak.com/realms/your-realm/protocol/openid-connect/token \
  -d "client_id=jobrunr-control" \
  -d "client_secret=${OIDC_CLIENT_SECRET}" \
  -d "grant_type=client_credentials" \
  | jq -r '.access_token')

curl -X POST \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  https://your-server/q/jobrunr-control/api/jobs/{jobId}/start
----

The service account must have the `api-executor` or `admin` realm role assigned in Keycloak.

==== Debugging Security Issues

To enable verbose security logging, add these properties scoped to your profile:

[source,properties]
----
%keycloak.quarkus.log.category."io.quarkus.security".level=DEBUG
%keycloak.quarkus.log.category."io.quarkus.oidc".level=DEBUG
%keycloak.quarkus.log.category."io.quarkus.vertx.http.security".level=DEBUG
----

Common symptoms and fixes:

[cols="2,2,2"]
|===
|Symptom |Cause |Fix

|401 on JobRunr Dashboard XHR calls (`/q/jobrunr/api/*`)
|OIDC tenant named `api` intercepts browser requests
|Rename the bearer tenant to `bearer`

|401/403 everywhere after login
|Roles not found — `role-claim-path` missing
|Add `quarkus.oidc.roles.role-claim-path=realm_access/roles`

|"You do not have access to the JobRunr Pro Dashboard"
|`JobRunrDashboardUserContextFilter` not registered
|Verify the extension is on the classpath

|`%dev.permit` active when using Keycloak
|Missing `%keycloak.*` policy overrides
|Add `%keycloak.quarkus.http.auth.permission.jobrunr-api-*.policy` overrides
|===

=== Default Values

Many parameters have default values:

* Pre-filled in the form when creating a new job
* Displayed as placeholder text in input fields
* Can be overridden by entering a custom value

=== Required vs Optional Parameters

* **Required** parameters are marked with an asterisk (*)
* Form cannot be submitted with empty required fields
* **Optional** parameters can be left empty

=== Parameter Storage

The extension supports two parameter storage strategies:

==== Inline Storage (Default)

Parameters are stored directly in JobRunr's job table as part of the job definition.
This is the default behavior for most jobs and suitable for:

* Jobs with small to medium parameter sets
* Jobs where parameters don't exceed typical database column limits
* Standard use cases

==== External Storage

For jobs with large parameter sets, parameters can be stored in a separate database table.
The job only stores a reference ID to the parameter set.

.When to Use External Storage
[cols="1,3"]
|===
|Scenario |Description

|**Large Text Fields**
|Jobs with multiline text parameters containing substantial content

|**Many Parameters**
|Jobs with 10+ parameters or complex data structures

|**Parameter Reuse**
|Multiple job executions sharing the same parameter set

|**Database Limits**
|Exceeding JobRunr's job metadata size limits
|===

[NOTE]
====
External storage requires a database connection to be configured in your Quarkus application.
You must manually create the required database table (`jobrunr_control_parameter_sets`) using the SQL scripts provided in the `docs/sql/` directory.
====

.Parameter Set Cleanup
The extension provides automatic cleanup of orphaned parameter sets:

* Configured via `quarkus.jobrunr-control.parameter-storage.cleanup.enabled`
* Default retention period: 30 days
* Runs periodically to remove unused parameter sets

[TIP]
====
Jobs using external storage are marked in the UI with a special indicator.
The dashboard transparently handles parameter loading and display.
====

Template jobs are reusable job configurations that serve as blueprints for creating and executing jobs on demand.

=== What are Template Jobs?

Template jobs are a special type of job that:

* Are **never executed directly** - they serve only as blueprints
* Are marked with a special "template" label
* Can be **cloned and executed** with optional parameter overrides
* Remain **unchanged** after execution (reusable)
* Are managed through a dedicated Templates tab in the dashboard

=== When to Use Templates

Template jobs are ideal for:

[cols="1,3"]
|===
|Scenario |Description

|**Recurring Operations**
|Jobs that run regularly with similar configuration (daily reports, weekly exports)

|**External Integrations**
|Jobs triggered by external systems with varying parameters

|**Customer-Specific Tasks**
|Same job logic executed for different customers or contexts

|**Parameterized Workflows**
|Complex workflows with configurable parameters per execution
|===

=== Executing a Template

==== Via UI (Admin Only)

[NOTE]
====
Requires **admin** role.
====

. Navigate to the **Templates** tab
. Find the template you want to execute
. Click the **Execute** button (play icon)
. A modal will appear allowing you to:
.. Enter a **postfix** for the job name (defaults to current date in yyyyMMdd format)
.. Override specific **parameters** if needed
. Click **Execute** to create and start the job

The system will:

. Clone the template
. Create a new job named: `{template-name}-{postfix}`
. Apply parameter overrides
. Execute the job immediately
. Display the execution in the Execution History

==== Via REST API

External systems can execute templates programmatically:

[source,bash]
----
POST /q/jobrunr-control/api/jobs/{templateId}/start
Content-Type: application/json

{
  "postfix": "20240127-customer123",
  "parameters": {
    "customerId": "123",
    "reportType": "DETAILED"
  }
}
----

Response:

[source,json]
----
{
  "jobId": "019c0021-a56a-73ef-9bd6-e74298078f97",
  "message": "Template job started successfully"
}
----

The returned `jobId` is the ID of the newly created and executed job (not the template).

=== Template Execution Flow

When a template is executed:

. **Template Retrieved**: The system loads the template configuration
. **Job Cloned**: A new job instance is created with the template's configuration
. **Name Generated**: Job name becomes `{template-name}-{postfix}`
. **Parameters Merged**: Default parameters + overrides
. **Job Scheduled**: New job is created as externally triggerable
. **Immediate Execution**: Job is enqueued and executed right away
. **Template Unchanged**: Original template remains available for reuse

=== Template Best Practices

==== Naming Conventions

* Use descriptive base names: `Customer-Report`, `Data-Import`, `Weekly-Cleanup`
* Avoid dates in template names (dates go in the postfix when executing)
* Use consistent naming across related templates

==== Parameter Configuration

* Set sensible **default values** for all parameters
* Use defaults that represent the most common use case
* Document expected parameter ranges in job descriptions

==== Postfix Usage

* Use date-based postfixes for time-based executions: `20240127`
* Use identifiers for entity-based executions: `customer-123`
* Combine both when needed: `20240127-batch-456`

==== Template vs Scheduled Job

[cols="1,2,2"]
|===
|Aspect |Template Job |Scheduled Job

|**Execution**
|Never executed directly, only via cloning
|Executed at scheduled time or via trigger

|**Reusability**
|High - creates new instances per execution
|Low - each execution updates the same job

|**Use Case**
|Recurring operations with varying parameters
|One-time or fixed schedule executions

|**Management**
|Managed in Templates tab
|Managed in Scheduled Jobs tab

|**Traceability**
|Each execution creates unique job
|Single job with execution history
|===

.Status Indicators
[cols="1,2"]
|===
|Status |Description

|**ENQUEUED**
|Job is queued and waiting to be processed

|**PROCESSING**
|Job is currently being executed

|**SUCCEEDED**
|Job completed successfully

|**FAILED**
|Job execution failed (may retry automatically)
|===

[NOTE]
====
**Job Chain Status Evaluation**

For jobs with continuation chains (using success or failure callbacks), the displayed status represents the **overall chain status**, not just the parent job:

* **Chain Complete**: All continuation jobs have finished (SUCCEEDED, FAILED, or DELETED)
* **Chain In-Progress**: At least one continuation job is still running (ENQUEUED, PROCESSING, PROCESSED)
* **Chain Success**: All executed continuation jobs succeeded
* **Chain Failure**: Any executed continuation job failed

This provides accurate visibility into complex job workflows.
Click the deep-link to JobRunr Pro Dashboard to see individual job details.
====

=== Filtering by Status

* All

The progress updates automatically while viewing the execution details.

== Executing Jobs

=== Execute Now (Admin Only)

[NOTE]
====
Requires **admin** role.
====

To immediately execute a scheduled job:

. Find the job in the Scheduled Jobs list
. Click the **Execute Now** button (play icon)
. The job will be moved to the processing queue immediately

=== Execute via REST API

External systems can trigger jobs using the REST API:

[source,bash]
----
# Trigger a job
POST /q/jobrunr-control/api/jobs/{jobId}/start

# Example with curl
curl -X POST http://localhost:8080/q/jobrunr-control/api/jobs/{jobId}/start \
  -H "Content-Type: application/json" \
  -d '{"param1": "value1", "param2": "value2"}'
----

== Handling Failed Jobs

=== Understanding Failures

Jobs can fail due to:

* Application errors (exceptions in job code)
* Infrastructure issues (database, network)
* Timeout exceeded
* Resource constraints

=== Automatic Retries

JobRunr Pro automatically retries failed jobs based on configuration:

* Default retry count is configurable per job type
* Exponential backoff between retries
* Jobs move to FAILED state after all retries exhausted

=== Manual Intervention

For failed jobs that need manual intervention:

. Navigate to JobRunr Pro Dashboard via deep link
. Review the error details and stack trace
. Use JobRunr's built-in requeue functionality to retry

=== Rerunning Failed Batch Child Jobs

For batch jobs with failed child jobs:

. Open the batch job in JobRunr Pro Dashboard
. Navigate to the failed child jobs
. Use JobRunr's batch management to retry failed items

The batch progress in JobRunr Control will update as child jobs complete.

== Tips and Best Practices

=== Naming Conventions

* Use descriptive job names that indicate purpose and context
* Include date or batch identifiers for recurring jobs
* Example: `Daily-Report-2024-01`, `Customer-Import-Production`

=== Parameter Management

* Use default values for commonly used configurations
* Document expected parameter formats in job descriptions
* Test parameter combinations before scheduling production jobs

=== Monitoring Recommendations

* Regularly check execution history for failed jobs
* Set up alerts in JobRunr Pro for recurring failures
* Review batch progress for long-running operations

=== Security Considerations

* Use role-based access control appropriately
* Limit admin access to authorized personnel
* Audit job executions through JobRunr Dashboard

== Audit Logging

All Template and Job operations are automatically logged for audit and compliance purposes.

=== Log Format

----
AUDIT [username] [action] [entity-type] [name] [id] [details]
----

=== Audited Operations

All create, update, delete, and execute operations for both templates and jobs are logged with:

* Authenticated username (or "anonymous")
* Full operation details including parameters
* Template/Job name and UUID
* For executions: source (REST/UI) and resulting job ID
